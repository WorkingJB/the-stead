# The Stead — PWA Architecture & Implementation Plan

> Approved plan for turning the authored Stead training programs into an installable, offline-first PWA. This document is the contract between the content authoring side of the repo (`content/`, `data/`, `shared/`) and the application code that will be added under `apps/web/`, `packages/content-pipeline/`, and `supabase/`.

## Context

The Stead is "a modern village in field clothes" with a fitness pillar that has already authored two complete 12-week training programs (Tier 0 bodyweight, Tier 1 standard home) plus four more planned tiers (2, 3, C1, C2). The programs live in `content/` (Markdown + YAML front-matter prose) and `data/progressions/<tier>.yaml` (tabular prescriptions). Each program has two fitness paths (A on-ramp, B build) and a fixed weekly cadence: strength A → Zone 2 → strength B → tempo/intervals → long easy. A movement library (push/pull/squat/hinge/core/locomotion) names regressions/progressions per movement.

**Goal:** ship those programs as an installable PWA so a user picks fitness level + equipment tier and gets a continuous, automatically-progressing workout series they can log sets/weights/RPE against — at the gym, offline, in dark mode.

- **Free tier:** browse any program and log workouts; swap lifts only from the curated movement-library list; last 4 weeks of history.
- **Paid tier:** customize lifts, edit schedule, export full history, per-movement analytics, multiple active enrollments.
- About-the-Stead content lives in-app.
- Security is a GTM differentiator, so it must be designed in from day one rather than retrofitted.

## Stack

- **Framework:** Next.js 15 (App Router, React Server Components) + TypeScript
- **Backend:** Supabase (Postgres + Auth + Storage + Row-Level Security)
- **Hosting:** Vercel + Supabase Cloud (region per user base)
- **State / data:** TanStack Query for server cache, Zustand for transient UI state
- **PWA:** `@serwist/next` (modern successor to next-pwa) for service worker + manifest
- **Offline DB:** Dexie (IndexedDB wrapper) for the local workout log + read-through cache of prescribed content
- **Styling:** Tailwind CSS v4 with design tokens from `shared/design-tokens.md` mapped to CSS variables; `next-themes` for dark mode; Radix UI primitives for accessible components
- **Forms / validation:** React Hook Form + Zod (shared schemas between client + server)
- **Payments (later):** Stripe via Supabase Edge Functions; entitlements stored on `profiles.tier_entitlement`

## Supabase schema (essentials)

User data lives in tables owned per-row by `auth.uid()`; canonical program content lives in tables readable by any authenticated user.

### Content (read-only to users; written by service role at build time)

- `tiers(id, slug, name, equipment_summary, description)`
- `movements(id, slug, name, pattern, body_md, alternatives jsonb)` — `alternatives` is the curated swap list per movement
- `programs(id, tier_id, length_weeks, paths text[])`
- `blocks(id, program_id, number, name, weeks int[])`
- `program_days(id, program_id, week int, day int, title, kind, duration_label, note, path text)`
- `prescribed_items(id, program_day_id, order_idx, movement_id nullable, raw_text, sets, rep_low, rep_high, duration_sec, load_hint)` — parsed from YAML strings so the UI can render structured logging fields while preserving raw text as a fallback

### User data (RLS = `user_id = auth.uid()`)

- `profiles(user_id pk, display_name, units, dark_mode, tier_entitlement text default 'free')`
- `enrollments(id, user_id, program_id, path, started_at, current_week, current_day, status)`
- `workout_sessions(id, user_id, enrollment_id, program_day_id, performed_at, rpe, sleep_prior, notes, body_weight)`
- `set_logs(id, session_id, prescribed_item_id nullable, movement_id, set_index, reps, weight, weight_unit, duration_sec, distance_m, completed bool)`
- `custom_programs(id, user_id, name, source_program_id nullable)` — paid tier
- `custom_program_days(...)` and `custom_prescribed_items(...)` — paid tier mirrors of canonical tables
- `audit_events(id, user_id, kind, payload jsonb, created_at)` — append-only

### Security primitives (first-principle)

1. **RLS on every user table**, default-deny. Policies: `using (auth.uid() = user_id)`. No service-role calls from the browser; only the seed script and Stripe webhook use service role.
2. **PII minimization.** Only `display_name` is collected. Email lives in Supabase-managed `auth.users`. No real names, addresses, or health conditions stored.
3. **At-rest encryption** of free-form notes via `pgsodium` symmetric column encryption keyed per-user — so even a leaked DB dump can't read notes without each user's key.
4. **Auth:** Supabase Auth with email magic-link + passkey/WebAuthn (no passwords). Session cookies are `HttpOnly`, `Secure`, `SameSite=Lax`; refresh handled by `@supabase/ssr`.
5. **Transport:** HSTS, strict CSP (no inline scripts; nonce-based), Permissions-Policy locked down, Subresource Integrity on third-party scripts, no third-party analytics by default.
6. **Audit log:** append-only `audit_events` written by Edge Functions for sensitive actions (export, account deletion).
7. **Right-to-delete:** one button → Edge Function that cascades deletes and revokes the Supabase user.
8. **Backups + threat model** documented in `SECURITY.md` and surfaced in-app under "About the Stead → Your data".

## Content pipeline (`packages/content-pipeline`)

A Node/TypeScript script run at build (and in CI) that:

1. Reads every `data/progressions/<tier>.yaml` and `content/<tier>/**/*.md`.
2. Parses each prescription string (e.g. `"Standard pushup: 3 × 8-12"`) into structured `prescribed_items` using a small grammar (movement name → lookup in movement library → sets, rep range, duration, load hint). Unparseable strings keep `raw_text` only.
3. Emits `supabase/seed.sql` + typed TypeScript constants used as the offline fallback bundle.
4. Validates referential integrity (every movement referenced exists; every week 1–12 present; every path has prescriptions).

Authors keep editing Markdown/YAML in git; the pipeline keeps the DB in lockstep. Shared Zod schemas between pipeline and app.

## PWA + offline strategy

- **Manifest + icons** generated from the brand palette in `shared/design-tokens.md` (forest `#2D4A35`, cream `#FBFAF4`).
- **Service worker (Serwist):** precache app shell + the user's current program (week ±1) + movement library; stale-while-revalidate for everything else.
- **Local-first writes:** every set log goes to Dexie first, then a background sync queue pushes to Supabase. Conflict policy: last-write-wins keyed on `(set_log.id, updated_at)`; IDs minted client-side as ULIDs so offline-created rows can be written directly.
- **Boot path:** on cold start, hydrate UI from Dexie immediately; revalidate from Supabase in the background. The gym screen must render within one frame even with no network.
- **Installability:** custom install prompt on `/about` once the user has logged one session.

## UI architecture

Routes (App Router):

- `/` — landing + about-the-stead summary, public
- `/about` — full "About the Stead" (renders from `content/<tier>/front-matter/*.md` server-side)
- `/onboarding` — 3 steps: fitness level (Path A/B with the existing "when in doubt, start on A" guidance) → equipment tier (cards per tier) → start date
- `/today` — the default authenticated landing; shows today's session big and legible
- `/program/[week]/[day]` — full week view; swipe between days
- `/log/[sessionId]` — the gym screen: huge tap targets, one card per prescribed item, weight/reps steppers, RPE slider, sticky "Save & next"
- `/history` — calendar + per-movement charts (free: last 4 weeks; paid: unlimited + export)
- `/movements/[slug]` — movement card with operator/longevity rationale, form cues, common errors, and **alternatives** (the curated swap list)
- `/settings` — profile, units, dark mode, data export, delete account
- `/billing` — Stripe customer portal (paid)

### Design-system mapping

- Tokens from `shared/design-tokens.md` become CSS variables in both light (cream background, ink text, forest brand, terracotta accent) and a dark theme (near-black background, cream text, muted forest/terracotta for accents to keep AA contrast).
- **Cormorant Garamond** for display, **Lora** for body, **Inter** for labels — loaded via `next/font` with subsetting; system font fallbacks for offline first paint.
- **Gym readability rules** (codified in Tailwind utilities): minimum body 18px, prescribed-set numbers at 32px+, tap targets ≥48×48, no pure-grey-on-grey, contrast checked in CI via `axe-core`.
- Radix primitives + `react-aria` hooks for screen-reader and keyboard parity; viewport breakpoints target 360→1280 with mobile-first layouts.

## Free vs paid entitlements

Single column `profiles.tier_entitlement` (`free` | `pro`) checked both in RLS policies (for write paths like `custom_programs`) and in route guards.

- **Free:** all tiers readable, full logging, alt-lift selection from the curated movement-library list, last 4 weeks of history.
- **Pro:** custom program builder, schedule editing, full history export (CSV + JSON), per-movement analytics, multiple active enrollments.

Stripe is wired but the upgrade CTA stays soft until paid features are actually built (phase 5).

## Phased delivery

**Phase 1 — Foundation (weeks 1–2):** Next.js + Tailwind + Supabase wired; auth (magic link); design tokens + dark mode; About page rendering from existing Markdown; CI with typecheck, lint, axe-core, Playwright smoke.

**Phase 2 — Content pipeline + read path (weeks 2–3):** `packages/content-pipeline` parses Tier 0 + Tier 1 into seed SQL; routes `/about`, `/movements`, `/program/[week]/[day]` render real prescribed content; full RLS migrations and `SECURITY.md`.

**Phase 3 — Logging + PWA (weeks 3–5):** onboarding flow, `/today`, `/log`, Dexie offline store + sync queue, service worker, install prompt. This is the MVP ship point — free tier complete.

**Phase 4 — History + analytics (weeks 5–6):** `/history` charts (Recharts), per-movement progression view, CSV/JSON export gated to Pro.

**Phase 5 — Paid customization (weeks 6–9):** custom program builder, schedule editor, Stripe + entitlements, audit log surfaced in `/settings`.

**Phase 6 — Remaining tiers:** as Tier 2/3/C1/C2 content gets authored in `content/` + `data/`, the pipeline picks them up with no app changes.

## Verification

Test plan executed at each phase boundary:

1. **Onboarding → today:** new account, pick Tier 0 + Path A, land on `/today` showing Week 1 Day 1 prescription parsed from `data/progressions/tier-0.yaml` (verify against the actual YAML lines).
2. **Offline log:** in DevTools, set network to offline, log all sets for the day, close tab, reopen → data still there; bring network back → row appears in Supabase within 10 s.
3. **RLS:** with two test accounts in two browsers, confirm account B cannot read or write account A's `set_logs` (direct REST call via Supabase JS).
4. **PWA install:** Lighthouse PWA audit ≥ 95; install on Android Chrome + iOS Safari; launch from home screen; complete a session offline.
5. **Accessibility:** `axe-core` 0 violations on `/today`, `/log`, `/about`; keyboard-only walkthrough of full log flow; VoiceOver and TalkBack pass.
6. **Dark mode + gym readability:** all text passes WCAG AA contrast in both themes; prescribed-number font-size ≥ 32px on mobile; tap targets ≥ 48px (Playwright visual check).
7. **Delete account:** triggers Edge Function, all `user_id`-scoped rows gone, `auth.users` row revoked, audit event written.
8. **Content fidelity:** generated seed for Week 6 Day 3 Path B exactly matches the YAML lines in `data/progressions/tier-1.yaml` (snapshot test in content-pipeline).
